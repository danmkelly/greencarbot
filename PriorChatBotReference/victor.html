The Business Challenge

Finding the right used car in a sea of listings

The used car market is fragmented across multiple sites: Craigslist, Autotrader, Facebook Marketplace, Cars.com, and dozens more. Buyers face thousands of listings with inconsistent descriptions, varying levels of detail, and no easy way to filter by the features that matter to them. A buyer looking for "a reliable SUV under $20k with good fuel economy and a clean title" might spend hours opening tabs and comparing spreadsheets.

The problem: buyers waste hours filtering manually: Is this car a good deal? Does it have the features I need? What's the average price for a 2018 Honda CR-V in my area? How does this Toyota compare to that Ford? These repetitive research tasks are a perfect use case for conversational AI.

The answer: a friendly, AI-powered car search assistant named Beep-Beep that knows every listing, every manufacturer, every feature, and every price point — straight from a curated Kaggle dataset of ~1200 clean used car listings.

This week's module concept is MCP (Model Context Protocol): giving an LLM access to a live data source so it can answer questions with real, up-to-date information rather than relying on training data alone. Beep-Beep Car Explorer fully implements this pattern.

The MCP data pipeline works as follows: the browser chatbot (AI client) sends a request to the Cloudflare Worker (/api/cars), which acts as the MCP server. The Worker fetches the Kaggle Craigslist Vehicles dataset (447K rows, preprocessed to ~1,200 clean listings with price, year, manufacturer, model, fuel, transmission, colour, mileage, and condition), filters it based on the user's query parameters (manufacturer, price range, year range, fuel type, transmission, body type, etc.), and returns a filtered JSON array. The chatbot then displays the matching cars in the results panel. No hard-coded data. No stale listings. One MCP data pipeline.

Either way, the architecture demonstrates the MCP pattern: the AI client talks to a data server, not to static files.

How Beep-Beep Works

AI-powered car search assistant, grounded in live data

🤖

Conversational AI

Beep-Beep is powered by DeepSeek, a large language model. It understands natural questions like "Find me a pickup truck under 100k miles with 4WD" and responds with accurate, data-backed results drawn from the live Kaggle dataset.

📊

Live Kaggle Dataset (MCP)

All ~1,200 car listings with price, specs, mileage, and condition data are fetched live from the Kaggle Craigslist Vehicles dataset via API. The chatbot queries the data in real time — no stale listings, no hard-coded inventory. This is the MCP pattern in practice.

🔍

Filter & Compare

Search results surface in an inline dual-panel view. The left panel shows matched listings as filterable cards; the right panel displays side-by-side comparison when two cars are selected. Buyers can compare specs, prices, and features in one glance.

Limitations & Known Weak Spots

Where Beep-Beep still struggles

Like any real-world AI project, Beep-Beep has rough edges. Here are the honest ones:

🔎

NLP Ambiguity

Beep-Beep uses keyword matching + LLM disambiguation to understand whether a user is asking about a specific model, a body type, a price range, or a comparison. Natural language is messy: "truck" might mean pickup, SUV-truck, or commercial vehicle. "Car" could exclude trucks entirely. The disambiguation gate asks clarifying questions when confidence is low, but ambiguous automotive terms remain challenging.

📊

Data Quality Gaps

The Kaggle Craigslist dataset is scraped data with inherent quality issues. Some listings have missing fields (no mileage, no VIN, placeholder descriptions). Duplicate listings from the same seller are de-duplicated heuristically but some may slip through. Price outliers (e.g., $1 listings for cars "call for price") have been filtered but the dataset reflects real Craigslist messiness.

⚠

Conversation Context

Beep-Beep maintains conversation history (last 10 exchanges) within the current session, and persists memory across visits using browser localStorage — including your name, recent searches, and favourited cars. On return visits, Beep-Beep greets you personally and references past searches. This memory is device-only: no data leaves your browser. A "Clear memory" button lets you wipe it at any time.

🎤

Voice Input Limitations

Voice recognition uses the browser's built-in Web Speech API, which varies in accuracy across browsers and devices. Car model names ("Cayenne", "Giulia", "Qashqai"), trim levels ("LX", "EX-L", "SEL Premium"), and technical terms ("all-wheel drive", "continuously variable transmission") are particularly prone to transcription errors. Voice output (TTS) may struggle with automotive terminology and numeric specs.

⏰

No Real-Time Inventory

The dataset represents a static historical snapshot from Kaggle, not live Craigslist listings. Cars marked as available may have sold months ago. Prices, locations, and availability do not update. This is appropriate for the academic scope (demonstrating MCP architecture) but a production system would need a live feed or web scraping pipeline.

💻

No Real Backend

Beep-Beep is a pure client-side application with no server-side database, authentication, or persistent favourites. The Cloudflare Worker proxy handles API key security, but all other state is browser-local. This is deliberate for the academic scope but limits production-readiness.

Technical Architecture

Secure, serverless, and key-safe

The LLM API (DeepSeek) requires a secret key. This is never exposed to the browser. A Cloudflare Worker acts as a thin proxy layer: the browser calls the worker, the worker injects the secret key server-side, and the response flows back. The Kaggle dataset is fetched via the Kaggle API (public endpoint for read access). No keys in source code, no keys in DevTools.

Development Approach

Built by a team of AI agent colleagues + MCP tools

The project was developed using a structured team of specialist AI agents, each with a defined role, domain expertise, and boundaries. This mirrors how a real innovation team would operate.

The team's design decisions are grounded in four academic best-practice frameworks from the Customer Engagement & AI module: the seven-dimension chatbot taxonomy (Adamopoulou & Moussiades 2020), CASA / Computers Are Social Actors (Reeves & Nass 1996), conversation flow anatomy, and Grice's maxims of conversation (Grice 1975). Each specialist agent owns the framework dimensions relevant to their domain.
