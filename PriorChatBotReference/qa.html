Design & Pedagogical Review

Academic framework assessment — Design Score: 4.0 / 5.0 — Grade B

Beyond the code-level bug register, this separate pass assesses Beep-Beep against four academic frameworks from the Customer Engagement & AI module. The review judges whether the bot is designed well as a conversational agent, not just whether it runs without errors. Keelin's final pass: all 22 findings closed (13 resolved, 9 Human Accepted). 0 FAILs. 12 PASS, 9 PARTIAL, 1 N/A. Design score: 4.0 (Grade B).

Classifies chatbots across 7 dimensions: knowledge domain, service provided, goals, response generation, communication channel, human-aid, and permissions. Beep-Beep is rated as a closed-domain (used car listings), task-based (car search + filtering + comparison), hybrid-generation (LLM + Kaggle data retrieval + keyword matching), text+voice bot with always-available human handoff. Key gaps: LLM hallucination risk (no RAG verification beyond Kaggle data), voice path may misrecognise car model names, trim levels, and technical terms.

Adamopoulou & Moussiades (2020). @Fionn owns domain closure & service scope. @Dot owns goals, response generation & channel. @Niamh owns human-aid reachability & permissions. @Siobhan ensures the UI makes each dimension visible.

Assesses whether the bot is CASA-AWARE (ethical, transparent) or CASA-BLIND (exploitative, deceptive). Beep-Beep has significant CASA protections: (1) disclosure in turn 1 is explicit ("I'm Beep-Beep, an AI car search assistant") -- PASS; (2) explicit consent flow asks permission before any localStorage write -- PASS; (3) frustration handling uses a gibberish counter with 3-step de-escalation ladder including human handoff -- PASS; (4) no fake-human cues (name is "Beep-Beep", a car horn sound, not a human name, showroom flavour retained as character seasoning) -- PASS. Register-vs-capability is PARTIAL as LLM knowledge of car valuations and market data is unverified.

Reeves & Nass (1996). @Siobhan owns disclosure-in-turn-1. @Dot owns frustration de-escalation as conversation engineering. @Niamh owns chummy-data-extraction consent. @Fionn verifies personality register matches actual capability.

Evaluates the structural completeness of Opener → Intent → Slots → Recovery → Closer. Beep-Beep passes on Opener (identity + capabilities in turn 1: "I can search for cars by make, model, body type, price range, or features") and Intent (clear purpose routing: car search, price filter, feature filter, car comparison). Recovery Ladder handles gibberish (3-step de-escalation), LLM errors (human path offered), empty Kaggle results (clarification offered). Slots now validated with make/model/body-type disambiguation when confidence is low -- PARTIAL (relies on LLM for entity extraction rather than structured form-filling). Closer offers "Refine search", "Compare selected", and "Human" suggestion chips after every search.

@Dot implements the recovery ladder in code (reword → options → human). @Siobhan designs the closer UI (refine + compare + human path). @Cormac uses "design the unexpected first" as a sprint-planning rule.

Tests Beep-Beep against Grice's four maxims of cooperative conversation (Grice 1975). An accurate bot can still feel wrong: grounding, scoped actions, and a clean handoff are what hold a bot to Grice.

Quantity -- say as much as needed, and no more. System prompt limits responses to 2-5 sentences (PASS). Quality -- only say what is true. Listing data comes from live Kaggle dataset (verified); LLM conversational text is unverified (PARTIAL). Relation -- be relevant. Intent routing is clear; post-search chips stay on-topic (PASS). Manner -- be clear, brief, and orderly. Responses are plain-language but prompt lacks explicit ordering rules (PARTIAL).

Grice (1975). @Dot owns Quantity & Quality (response length, LLM grounding). @Siobhan owns Manner (clarity, brevity in UI). @Fionn owns Relation (domain-relevance guardrails). @Keelin cross-references Grice alongside Taxonomy, CASA, and Flow Anatomy.

Testing Methodology

Functional testing — Overall Score: 3.5 / 5.0 — Grade B-

All testing follows a real-user simulation model: tests are written as if a car buyer on a Sunday afternoon is trying to find the right used car. No test assumes a happy path. Every test includes a verification step that can be automated using a headless browser or simulated DOM interaction.

The test plan covers 11 categories:

  • Greeting flow -- first visit, return visit, name capture
  • Car search queries -- by make, by model, by body type, combined
  • Filter combinations -- price + year + mileage + transmission + fuel type
  • Price range searches -- under budget, between range, best deals
  • Car comparisons -- dual-panel side-by-side, spec highlighting
  • Kaggle data loading (MCP) -- fresh fetch, cache, error handling
  • Human handoff -- contact details, dealer referral
  • Memory/persistence -- localStorage, clear, return visit, favourites
  • Voice STT -- mic, transcription, auto-send
  • Voice TTS -- speaker icon, playback, cancel
  • Mobile -- breakpoints, layout, touch targets

Each test case specifies what DOM element to check, what text to assert, and what network call to verify. Tests are designed to be run by a headless browser script (Puppeteer/Playwright) that:

  • Sets the viewport to the target device width
  • Simulates clicks, typing, and voice input
  • Asserts DOM state after each interaction
  • Captures screenshots on failure
  • Logs console errors as test failures
